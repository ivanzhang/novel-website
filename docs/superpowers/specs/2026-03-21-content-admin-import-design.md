# 后台内容管理与批量导入分类设计

## 目标

为当前中文小说阅读站增加一个面向单人运营的后台内容管理模块，重点解决以下问题：

- 将抓取到的原始 HTML 批量导入系统
- 自动抽取小说基础信息与章节数据
- 通过规则完成主分类与标签推荐
- 在上架前拦截疑似重复内容
- 以最少人工操作完成审核与发布

本设计的核心不是传统 CMS 式手工录入，而是围绕“导入、整理、审核、上架”的运营工作流构建一个内容导入工作台。

## 背景与范围

已确认的前提如下：

- 第一阶段由单人使用
- 内容规模先控制在几百本小说
- 导入源以盗版站抓取的原始 HTML 为主，结构化数据为辅
- 分类策略采用“规则优先、人工校正、后续可接 AI”
- 默认分流策略为：低风险内容快速上架，高风险或异常内容进入待审核
- 风险判断中，第一优先级是疑似重复或撞库

第一阶段只设计后台内容管理与导入分类相关能力，不包含：

- 多人协作权限系统
- 复杂经营报表
- AI 主分类能力
- 大规模内容平台级任务调度
- 富文本手工编辑器

## 成功标准

满足以下条件即可视为第一阶段达到目标：

- 可以创建批量导入任务并导入原始 HTML、ZIP 或结构化 JSON
- 大多数小说能自动得到可用的主分类与标签建议
- 疑似重复内容会在进入正式书库前被拦截
- 导入失败项可以单本重试，不影响整批
- 运营者可以通过后台批量审核并上架内容

## 方案选择

对比过的方案如下：

### 方案一：轻后台录入型

以传统后台表单为中心，支持手工新建小说、上传章节、手工分类和上架。

优点：

- 实现最简单
- 页面直观

缺点：

- 无法解决批量导入与清洗的核心痛点
- 长期会把运营时间浪费在重复手工录入上

### 方案二：导入管道优先型

以导入任务和待处理书库为中心，核心流程为“导入 -> 解析 -> 分类 -> 去重 -> 审核 -> 上架”。

优点：

- 最贴合当前的单人运营与批量导入诉求
- 能把人工操作压缩到必要的审核决策
- 后续可平滑扩展分类规则和数据源

缺点：

- 需要先设计状态流转和 staging 数据区

### 方案三：内容平台型

一步到位做来源管理、审核流、权限、日志、批处理和完整运营平台。

优点：

- 扩展空间最大

缺点：

- 对当前几百本、单人运营的阶段明显过重
- 会延迟最核心的导入效率问题

### 结论

采用方案二：导入管道优先型。

## 总体设计

后台新增一个轻量管理模块，整体分为两层数据区：

- `staging`：存放原始抓取数据、解析结果、分类结果和审核状态
- `production`：正式对前台可见的小说与章节数据

强约束：

- 导入逻辑不能直接写入正式 `novels` 和 `chapters`
- 所有导入内容都必须先进入 staging 区
- 只有通过必要审核的内容才能转入正式书库

这样可以在解析规则、分类规则或去重逻辑发生变化时重跑处理流程，而不污染正式数据。

## 核心数据模型

第一阶段建议新增或扩展以下实体：

### `import_jobs`

表示一次批量导入任务，例如“从某来源导入 50 本小说”。

建议字段：

- `id`
- `source_name`
- `input_type`：`url_list`、`html_zip`、`json`
- `status`
- `total_count`
- `success_count`
- `failed_count`
- `pending_review_count`
- `created_at`
- `updated_at`

### `import_items`

表示导入任务中的单本小说，是后台处理的最小单元。

建议字段：

- `id`
- `job_id`
- `source_record_id`
- `raw_payload`：原始 HTML 或原始 JSON
- `parsed_title`
- `parsed_author`
- `parsed_intro`
- `parsed_status`
- `parsed_cover_url`
- `parsed_chapter_count`
- `normalized_payload`
- `classification_result`
- `duplicate_score`
- `duplicate_candidates`
- `status`
- `failure_reason`
- `created_at`
- `updated_at`

### `source_records`

用于记录来源与追踪信息。

建议字段：

- `id`
- `site_name`
- `source_url`
- `source_category_name`
- `fetched_at`
- `source_hash`

### `categories`

站内主分类表。

### `tags`

站内标签表。

### `novel_aliases`

用于保存一本正式小说的别名，支撑后续重复判断。

建议字段：

- `id`
- `novel_id`
- `alias`
- `source_name`
- `created_at`

### `duplicate_candidates`

可选表。用于缓存疑似重复的评分结果和审核动作，避免重复计算。

如果实现量需要压缩，第一阶段最少应保证：

- `import_jobs`
- `import_items`
- `novel_aliases`

## 导入入口

第一阶段只做 3 类导入入口：

### 粘贴来源链接

适合临时导入少量目标。

### 上传 HTML 或 ZIP

适合已经完成抓取、需要最快进入后台处理的数据。

### 导入结构化 JSON

虽然当前不是主入口，但必须预留，以便后续爬虫直接输出标准化数据。

所有入口都应统一落到 `import_jobs` 和 `import_items` 上，不为不同输入方式设计完全不同的数据流。

## 数据保留原则

每本导入内容都必须保留两份数据：

### 原始数据

- 原始 HTML 或原始 JSON
- 原始 URL
- 来源站点
- 原始栏目
- 抓取时间

### 标准化数据

- 标准化后的书名
- 作者
- 简介
- 封面
- 章节列表
- 主分类建议
- 标签建议
- 当前状态

这样当解析规则升级时，可以重新解析而不必重新抓取。

## 状态流转设计

每本导入内容建议采用以下固定状态：

1. `collected`
2. `parsed`
3. `pending_classification`
4. `pending_duplicate_review`
5. `pending_publish`
6. `published`
7. `failed`

状态说明：

- `collected`：原始数据已入库，但尚未解析
- `parsed`：基础字段和章节已抽取完成
- `pending_classification`：等待确认自动分类结果
- `pending_duplicate_review`：存在疑似重复，需要人工判断
- `pending_publish`：已满足上架条件，等待最终确认
- `published`：已进入正式书库
- `failed`：解析失败或数据异常

约束：

- 状态转换必须可追踪
- 单本失败不能阻塞整批任务
- 允许对失败项单本重试

## 分类体系设计

第一阶段采用三层结构：

### 主分类

每本书只能有一个主分类，建议初始集合如下：

- 玄幻
- 仙侠
- 都市
- 言情
- 历史
- 科幻
- 游戏
- 悬疑
- 同人
- 其他

### 标签

每本书可有多个标签，例如：

- 系统流
- 重生
- 穿越
- 无敌
- 后宫
- 校园
- 总裁
- 末世
- 种田
- 直播

### 来源属性

不直接作为前台分类展示，但后台必须保留：

- 来源站点
- 原始栏目
- 是否完结
- 是否 VIP
- 抓取时间

## 自动分类规则

第一阶段采用规则优先的半自动分类，不引入 AI 主分类。

执行顺序如下：

### 1. 来源栏目映射

根据来源站原始栏目映射站内主分类。例如：

- `玄幻奇幻` -> `玄幻`
- `武侠仙侠` -> `仙侠`
- `都市言情` -> `都市` 或 `言情`

### 2. 书名关键词

若来源栏目不足以确定，则根据书名关键词补充判断。

### 3. 简介关键词

当栏目与书名命中不足时，用简介关键词给出推荐。

### 4. 人工修正规则回写

当运营者手工修正某一类映射时，系统应允许将该修正沉淀为新规则，供后续复用。

第一阶段交互要求：

- 自动分类结果必须展示命中依据
- 支持按来源栏目批量确认主分类
- 支持补充和批量设置标签

## 重复判断设计

重复判断目标是高召回发现疑似重复，再用低成本人工确认收口。

第一阶段不做自动合并正式内容。

### 判重信号

#### 强信号

- 书名完全一致
- 标准化书名完全一致
- 书名一致且作者一致

书名标准化至少应包括：

- 去除空格
- 统一全角半角
- 去除常见噪音词，如“全文阅读”“最新章节”“无弹窗”“笔趣阁”
- 统一标点与括号差异

#### 中信号

- 书名近似
- 作者近似
- 章节数接近
- 简介前若干字相似

#### 辅助信号

- 来源站点相同
- 原始栏目相同
- 首章标题相同
- 末章标题相同

### 判重结果分级

- `high`：高疑似，优先进入重复审核列表
- `medium`：中疑似，展示候选但不阻塞全部流程
- `low`：低疑似，仅记录

### 审核动作

后台只提供 3 个动作：

- `作为新书保留`
- `合并到已有书`
- `稍后处理`

第一阶段的“合并”仅包含：

- 选择已有正式书作为主记录
- 可选补充新导入中缺失的章节
- 保留来源信息和别名

不做复杂字段级自动融合。

## 风险分流

导入后的内容默认按风险分流：

- 低风险且规则命中充分的内容进入 `pending_publish`
- 高风险、数据异常或存在高疑似重复的内容进入审核队列

第一阶段最重要的风险判断维度是重复或撞库。后续可以继续引入：

- 来源站点等级
- 数据完整度
- 文本质量

## 后台页面设计

第一阶段只做 6 个页面：

### 1. 工作台

默认首页，不做装饰性报表，直接展示待办：

- 待处理导入任务
- 导入失败数
- 待分类数
- 待重复审核数
- 待上架数
- 最近已上架内容

### 2. 导入任务页

功能：

- 新建导入任务
- 上传 HTML、ZIP、JSON
- 查看批次进度
- 查看成功、失败、待审核数量
- 失败条目重试

### 3. 待处理书库

作为日常主工作页，列表建议字段：

- 书名
- 作者
- 来源站
- 章节数
- 自动分类结果
- 重复风险等级
- 当前状态
- 最后更新时间

必须支持：

- 状态筛选
- 来源筛选
- 批量操作
- 侧边预览抽屉

### 4. 分类处理页

用于高效确认自动分类结果，重点是批量确认而不是深度编辑。

### 5. 重复审核页

并排展示新导入书与已有书信息，以及命中依据和操作按钮，降低人工决策成本。

### 6. 正式书库

用于已上架内容的基础维护，如搜索、下架、修正主分类和标签。

## 日常操作流

建议后台围绕以下固定工作流设计：

1. 打开工作台查看待办
2. 优先处理导入失败项
3. 批量确认高置信分类结果
4. 处理高疑似重复内容
5. 在待上架列表中批量确认发布

后台设计应围绕这一主路径优化，而不是先追求复杂报表或大而全功能导航。

## 错误处理与重试

导入失败原因必须显式化，至少包括：

- 未识别书名
- 作者为空
- 简介缺失
- 章节提取为 0
- 正文疑似乱码
- HTML 结构不匹配
- 疑似与已有书重复

失败条目必须支持：

- 查看原始内容
- 调整规则后重新处理
- 单本重试

禁止要求整批重新执行才能恢复单本失败。

## 测试与验证

第一阶段至少需要覆盖以下验证：

### 导入解析测试

使用典型 HTML 样本验证书名、作者、简介、章节提取是否正确。

### 分类规则测试

验证来源栏目映射、关键词命中和规则回写是否按预期工作。

### 判重测试

覆盖同书不同名、简繁差异、作者近似、章节数接近等情况。

### 状态流转测试

验证导入内容不会绕过 staging 和审核直接进入正式书库。

### 批量操作测试

验证批量分类、批量上架和失败重试流程。

## 非目标

以下内容不纳入第一阶段：

- 多角色权限与审批流
- 高级操作日志和审计系统
- AI 自动摘要、AI 主分类、AI 去重
- 分布式队列与超大规模调度
- 复杂推荐系统和经营分析后台

## 实施建议

如果需要进一步压缩第一阶段范围，建议优先实现以下最小闭环：

- 导入任务
- 待处理书库
- 自动分类基础规则
- 疑似重复候选列表
- 批量上架

在此基础上先把“导入整理”主路径打通，再逐步增强来源治理、规则沉淀和更复杂的内容管理能力。
